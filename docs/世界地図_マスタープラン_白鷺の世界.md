# 播州サバイブ 世界地図 — マスタープラン「白鷺の世界」

**版:** 2026-07-09（iPad対応 P-T を追加） / 対象: `js/world-map-editor.js`（v81・5,595行）
**使い方:** 本書はCursorへの指示の親玉。各フェーズの「Cursorプロンプト」をそのままコピペして進める。フェーズは独立しており、1フェーズ＝Cursorの1〜3セッションを想定。

## 0. ビジョン（憲法）

読者は地図を「見る」のではない。**白鷺としてこの世界に入る。**
高く飛べば播磨が霞み、降りれば喫茶店の看板が読める。
世界は作者のものではない——他者の世界に客人として降り、出会ったものに印と言葉を残す。

### 三原則（すべての実装判断はこれに従う）

1. **正直なメートル** — すべての物体は実寸（メートル）で定義し、`metersToUnits()` で配置時に変換する。車は4.5m、電柱は12m、人は1.7m。誇張しない。スケール感はサイズの水増しではなく「参照物・霞・視差」で作る。
2. **地形は遠景、プロップは近景** — グリッドsculpt地形は「背景の大地」であり、これ以上解像度を追わない（凍結）。近くで見る密度は、配置される素材（プロップ）が担う。
3. **エンジンとワールドの分離** — 播州固有のものはすべてデータ（Supabase行・設定オブジェクト）に置き、コードには置かない。これが将来「静岡の誰か」が使える形への布石。
4. **タッチファースト（iPadが主編集デバイス）** — 世界描写は指・Apple Pencilで行うのが本流。新規入力コードはすべて Pointer Events で書き、マウスは「動くのが当然の副次デバイス」として扱う。片手（1本指）＝描く、2本指＝世界を動かす、を全ツール共通の文法にする。

### なぜこれで3つの問題が解けるか

| 問題 | 原因（コード上の事実） | 解 |
|------|------|------|
| サイズ感・ズームが合わない | カメラはorbit距離ベースで地表に「立てない」。白鷺はDOM画像の投影（`updateShirasagiAvatar`）で世界の中にいない。物を50倍に水増しして補っている | **鳥モード**：高度に比例した移動速度・実寸プロップ・near平面0.1（既に`PerspectiveCamera(48, .., 0.1, 12000)`）で、実寸の車のそばまで降りられる |
| 物を一個ずつコードで建てる地獄 | `InstancedMesh`ゼロ。高架・堤防等すべて専用コード | **播州キット**：パラメトリック素材レジストリ＋InstancedMesh。一度定義したら「置くだけ」 |
| 時間を食われ、スポット登録が遅れる | 編集＝建設作業になっている | 建設をキット化で終わらせ、以後の編集を「配置＋言葉」＝コンテンツ作業に転換 |

### 予算（守るべき性能の枠）

- 地形: seg480＝約23万頂点（現状維持、増やさない）
- プロップ: 1ゾーンあたりインスタンス合計 5,000 まで／draw call はキット種別ごとに1（InstancedMesh）
- テクスチャ: 原則使わない（頂点色＋単色マテリアル）。ローポリ＋霞で世界観を作る
- 対象: ノートPCのブラウザで60fps、スマホで30fps

## P-T. タッチ入力基盤 — iPadを主編集デバイスにする

**目的:** 世界描写（sculpt・塗り・配置）は指・Apple Pencilのほうが正確で速い。iPadを**主**編集デバイスに、マウスを副に位置づけ直す。以後のフェーズの入力はすべてこの基盤の上に書く。

### コード上の現状（事実）

- キャンバス操作は `mousedown`（4477行）/ `mousemove`（4563行）/ `mouseup`（4622行）の**マウス専用**。iPad の Safari では sculpt が一切動かない
- タッチ対応済みなのは D-pad ボタン（`touchstart`/`touchend`、4846行付近）のみ
- ズームは `wheel` イベント頼み（ピンチなし）
- CSS に `touch-action` 指定なし → タッチするとページスクロール・ダブルタップズームが暴発する
- UIは `:hover` 前提・ボタンが小さい箇所がある

### ジェスチャ文法（全ツール共通・Procreate流）

| 入力 | 動作 |
|------|------|
| 1本指ドラッグ | **選択中ツールを実行**（掘る・塗る・置く。lookツール時は回転/パン） |
| Apple Pencil | 常にツール実行（指より優先。パームリジェクション） |
| 2本指ドラッグ | パン（平面）/ 軌道回転（立体） |
| 2本指ピンチ | ズーム（wheel相当） |
| 2本指タップ | Undo |
| 3本指タップ | Redo |
| 長押し | 右クリック相当（削除等・P2で使用） |

**原則:** 「1本指＝描く、2本指＝世界を動かす」。モード切替ボタンで描画/移動を切り替えさせない（それは失敗パターン）。

### 仕様

- `js/world/input-touch.js` を新設。export: `attachPointerInput(dom, handlers)` — 既存 mouse リスナーを **Pointer Events に置き換え**、`handlers = { onStrokeStart/Move/End, onPan, onOrbit, onPinch, onTwoFingerTap, onThreeFingerTap, onLongPress }` に正規化して渡す
- `setPointerCapture` を使い、キャンバス外に指が出ても stroke が切れないように
- `pointerType === 'pen'` は常に stroke。指は「アクティブポインタが1本なら stroke、2本目が着地したら stroke をキャンセルして gesture へ昇格」（誤爆防止のため着地後 ~80ms は判定猶予）
- ピンチは2本指間距離の比で `orbit`/`planZoom` を更新（既存の wheel 処理を関数化して共用）
- キャンバスに `touch-action: none`。`<meta name="viewport">` に `maximum-scale=1, user-scalable=no` を追加（誤ズーム防止。アクセシビリティ上の懸念よりも編集ツールとしての操作性を優先）
- UI調整（`css/world-map.css`）: `@media (pointer: coarse)` でツールボタン・±ステッパーの最小ヒット領域を 44×44px に、`:hover` 依存の表示は `:active`/選択状態で代替
- 既存のマウス操作（Shift+ドラッグでパン等）は**そのまま動くこと**（Pointer Events はマウスも統一的に扱える）
- D-pad の既存 touch 対応はそのまま活かす

### 受け入れ基準

iPad Safari（またはMacのトラックパッド偽装ではなく実機）で:
1. zone-4 を開き、指で「地面を盛る」「川を掘る」が**線として**なめらかに描ける
2. 2本指でパン・ピンチズーム・（立体で）回転ができ、描画と混線しない
3. 2本指タップで Undo できる
4. ページ全体がスクロール・ズームしない
5. Mac のマウス操作が従来どおり動く

### Cursorプロンプト


## P1. 空気 — 霧・空・遠近感

**目的:** 「見上げる山の高さ」を出す。スケール知覚の最重要要素は空気遠近（遠いものが霞む）であり、現在 `scene.fog = null`（291行目）で完全に切られている。ここを入れるだけで世界の奥行きが一変する。

**仕様:**
- `js/world/atmosphere.js` を新設。export: `initAtmosphere(scene, renderer)`, `updateAtmosphere(cameraAltMeters, timeOfDay)`
- `THREE.Fog` を導入。near/farはワールドサイズ比で設定（例: near=SIZE*0.15, far=SIZE*1.1）。**高度が上がるほどfarを伸ばす**（高く飛ぶと遠くまで見えるが、より霞む）
- 空: 背景色を単色からグラデーションに（大きな逆向き球 or `scene.background` の縦グラデCanvasTexture）。地平線付近を霞色（fog色と一致させる）に
- 時刻パラメータ `timeOfDay`（0-1）で空とfogの色相を朝・昼・夕・夜に補間（自動進行はオフでよい、まず手動スライダー）
- 平面ビュー（`viewMode === 'plan'`）ではfog無効
- 色はプリセットごとにデータで持つ（三原則3）: `presets[].atmosphere = { fogColor, skyTop, skyBottom }`

**受け入れ基準:** コヌイzone-4の立体ビューで、遠くの地形の稜線が霞に溶け、山が「遠く・大きく」感じられる。fpsが落ちない（fogはほぼ無料）。

**Cursorプロンプト:**
```
js/world/atmosphere.js を新設し、世界地図の立体ビューに空気遠近を実装してください。

前提:
- world-map-editor.js 291行目付近で scene.fog = null になっている
- viewMode 変数が 'plan' | '3d' を持つ
- SIZE がワールドの一辺（ワールド単位）
- perspCamera は PerspectiveCamera(48, aspect, 0.1, 12000)

要件:
1. initAtmosphere(scene) と updateAtmosphere({ altUnits, timeOfDay }) をexport
2. THREE.Fog(color, SIZE*0.15, SIZE*1.1)。高度altUnitsが上がるほど
   far を最大 SIZE*2.0 まで線形に伸ばす
3. scene.background を縦グラデーション（CanvasTexture 2x256px）にし、
   下端色は fog 色と一致させて地平線を溶かす
4. timeOfDay (0-1) で fog色/空色を [朝, 昼, 夕, 夜] の4キーカラー間で補間。
   色セットはプリセット定義（presets）に atmosphere オブジェクトとして追加し、
   コードにハードコードしない
5. viewMode==='plan' では scene.fog=null に戻す
6. world-map-editor.js 側の変更は import と、レンダーループ・ビュー切替での
   呼び出し追加のみ。既存機能を壊さない
7. 管理画面の表示パネルに timeOfDay スライダー（0-1）を1本追加
```

## P2. 播州キット — プロップシステム基盤

**目的:** 「物を一個ずつコードで建てる」の終了。素材（キット）を実寸パラメトリックで一度定義し、以後は**置くだけ**。Minecraftから盗むのは構造：素材が先にあり、世界は配置の集積。

### データモデル

Supabase新テーブル（SQLはそのままCursorに渡す）:

```sql
create table if not exists map_props (
  id uuid primary key default gen_random_uuid(),
  world_id text not null,
  layer_id text not null,          -- 既存の activeLayerId() と同じ体系
  kind text not null,              -- 'house' | 'kissaten' | 'pole' | 'bridge' ...
  x double precision not null,     -- 正規化 0-1（既存スポットと同じ流儀）
  z double precision not null,
  rot double precision default 0,  -- ラジアン
  params jsonb default '{}',       -- 種別ごとの寸法・色（すべてメートル）
  label text,                      -- 任意の名前（「摩天楼」等）
  spot_slug text,                  -- 既存 map_spots への任意リンク
  created_at timestamptz default now()
);
create index on map_props (world_id, layer_id);
```

localStorageは既存の短縮キー方式に `p`（props配列）を足す。

### キット v1（この6種で始める。増やすのは動いてから）

すべて実寸メートル・単色マテリアル・ローポリ（各50ポリゴン以下目安）:

| kind | 内容 | 主要params（デフォルト） |
|------|------|------|
| `house` | 切妻屋根の家 | w:8, d:6, h:5.5, roofColor, wallColor |
| `kissaten` | 家＋前面庇＋看板板 | w:10, d:8, h:4.5, signColor |
| `pole` | 電柱＋腕金 | h:12 |
| `bridge` | 桁橋（板＋両端の橋台） | len:30, w:6, deckH:4 |
| `levee` | 堤防セグメント（台形押し出し） | len:40, topW:4, h:5 |
| `torii` | 鳥居 | h:6 |

**電柱は必ず入れる。** 12mの電柱が立っているだけで、人間はその世界のスケールを一瞬で読む。田園の道に電柱が並ぶだけで「日本の郊外」になる。参照物こそがサイズ感の解。

### アーキテクチャ

- `js/world/props.js`: レジストリ方式。`registerKind(kind, buildGeometry(params))` → 種別ごとに `InstancedMesh`（同一マテリアル）へ集約。draw callは種別×マテリアル数のみ
- 配置はグリッドに縛らない（連続座標）。高さは地形サンプリング（既存の地形高さ取得を利用）で自動接地
- 配置ツール: 既存ツールバーに「置く」を追加。種別選択→クリックで設置、ドラッグで回転、右クリック/長押しで削除。Undo対象に含める
- 保存: `saveAll()` にpropsのupsert/delete追加。読込: `loadFromSupabase()` 後にprops取得→一括インスタンス化

**受け入れ基準:** zone-4に家20軒・電柱30本・橋1本を10分以内で置けて、保存→リロードで復元され、fpsが落ちない。

**Cursorプロンプト（基盤）:**
```
js/world/props.js を新設し、パラメトリック素材（プロップ）システムを実装してください。

設計原則:
- すべての寸法は実寸メートルで定義し、配置時に metersToUnits() で変換
- 種別ごとに THREE.InstancedMesh に集約（1種別＝1〜3 draw call）
- 播州固有の情報（色・寸法のデフォルト）は KIND_DEFS データオブジェクトに置く

実装:
1. registerKind(kind, def) / buildAllInstances(propsArray) / 
   raycast対象の取得 をexport
2. KIND_DEFS に house / kissaten / pole / bridge / levee / torii の6種。
   各 def = { params(デフォルト寸法m), build(params)→BufferGeometry(複数
   マテリアル可), footprint }
3. ローポリ・頂点色または単色。テクスチャ禁止
4. 接地: 配置座標の地形高さを取得して y を決定（既存の高さサンプリング
   関数を再利用）
5. world-map-editor.js への統合: ツール 'prop' を追加、種別セレクタUI、
   クリック設置・ドラッグ回転・削除、Undo履歴への組み込み
6. 保存/読込: localStorage短縮キー 'p'、Supabase map_props テーブル
   （SQLは docs に置く）。saveAll / loadFromSupabase を拡張
7. 既存の樹・スポットの仕組みは変更しない
```

## P3. 白鷺モード — 鳥として世界に入る

**目的:** ビジョンの核。「俯瞰」から「没入」へ。読者は編集カメラではなく、**白鷺の身体**で世界を飛ぶ。

### 設計

- `js/world/bird.js`: 三人称の鳥コントローラ
- **鳥の実体:** ローポリ白鷺（胴・首・頭・翼2枚、〜80ポリ）。翼は `sin(t)` で羽ばたき、滑空時は水平固定。DOM画像の白鷺（`updateShirasagiAvatar`）は鳥モード中は非表示
- **操作:** 既存の D-pad/W/S/Q/E を流用。前進は自動（ゆっくり滑空）でもよいが、まずは既存キーで前後左右・高度・旋回。**タッチ（P-T基盤）:** 1本指ドラッグ＝視線・旋回、2本指上下＝高度、ピンチ＝速度（または高度）。D-padはタッチでも既に動くため初期はD-padのみで可
- **高度比例の速度（最重要）:** `speed = baseSpeed * (1 + altMeters / 15)`。地上2mではそろそろと車の脇を進み、高度300mでは町を一息で渡る。これが「ズームの比率が合わない」問題の運動学的な解
- **カメラ:** 鳥の後方やや上に追従（ばね補間）。near=0.1のまま。地面すれすれ（最低高度=地形+1.5m）まで降りられる
- **既存カメラとの関係:** 「編集カメラ（現行orbit）」と「白鷺モード」をトグル。公開ページ（readOnly）は**白鷺モードをデフォルト**にする
- P1の霧と連動: 高度で `updateAtmosphere` を呼ぶ。低空では近くだけ見え、上昇すると世界が開ける——これ自体が演出

**受け入れ基準:** zone-4で、上空から降下→実寸の家々の屋根すれすれを飛び→電柱の横を抜けて着地近くまで降りる、が気持ちよくできる。「見上げると山（地形の高まり）が大きい」と感じられる。

**Cursorプロンプト:**
```
js/world/bird.js を新設し、白鷺として世界を飛ぶ三人称モードを実装してください。

前提:
- 既存に DOM画像の白鷺投影（updateShirasagiAvatar）と heronAltUnits() がある
- カメラは perspCamera（near 0.1）、操作は D-pad + Q/E回転 + W/S高度
- metersToUnits() で実寸変換、地形高さサンプリング関数が既存

要件:
1. ローポリ白鷺メッシュ（胴・首・頭・翼×2、80ポリ以下、白＋くちばし黄）。
   翼は羽ばたきアニメ（速度が高いほど速く、降下時は滑空で固定）
2. モード切替 setBirdMode(on)。鳥モード中は DOM白鷺を非表示、
   編集ツールを 'view' 相当に
3. 移動速度を高度に比例させる: speed = base * (1 + altMeters/15)。
   最低高度 = 地形高さ + 1.5m（めり込み禁止）
4. カメラは鳥の後方上方にばね追従（lerp 0.08程度）、旋回Q/Eで鳥ごと回る
5. P1 の updateAtmosphere({altUnits}) を毎フレーム呼ぶ
6. 公開ページ（readOnly: true）では鳥モードをデフォルトON、
   管理画面ではトグルボタンで切替
7. 既存の編集カメラ・平面ビューの動作は不変
```

## P4. 出会いの接続 — スポット→物語カード

**目的:** 世界を飛んでいて、プチダイヤモンドの上に来ると印があり、触れると言葉が立ち上がる。「一覧を読む」ではなく「さまよって出くわす」。

## P5. スケール間ダイブ — 3世界の接続演出

**目的:** ニューハリマ（広域）→ヒメモリの地→コヌイの路、を「ページ切替」ではなく「降下」として体験させる。世界の重ね合わせの可視化。

**仕様（演出で解く。真のシームレスLODストリーミングは作らない——個人開発の予算外）:**
- 広域ワールドに「降下点」マーカー（コヌイの位置に光の柱など）を置く
- 白鷺で降下点上空に入り一定高度以下へ降りると、①霧が濃くなりホワイトアウト → ②次のワールドを裏で読込 → ③高高度から霧が晴れながら出現
- 逆方向（上昇で広域へ）も同様
- 実装は既存のプリセット切替（`orbit = preset().orbitDefault` 等）をラップするトランジション関数

**受け入れ基準:** 播磨全域から2回の降下でzone-4の電柱の高さまで、読み込み画面を感じずに降りられる。
